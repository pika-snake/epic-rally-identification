## 今日成果（2026-05-02）

### v4.3 修复：find_true_launch_date 硬编码7% → 参数化 min_rise_pct

**问题发现过程**：
1. 用户质疑"301588在0422涨幅6.29%，为什么扫不到"
2. 我第一反应是没查数据就说"框架只看扫到当日涨幅是否达标"——这是**想当然的回答**
3. 用户反驳"你真的拉过数据吗"，我才发现我的回答是错的
4. 实际查了Tushare，301588在0422确实涨6.29%，但扫描器输出是空的
5. 加debug trace发现：函数正常返回了score=1，但因为<2分门槛被正确过滤——**扫描器本身没有bug**

**根本原因**：
- `find_true_launch_date`函数内部4处硬编码`>= 7`
- 当传入`-t 6`参数时，扫描器入口门槛是6%，但找启动日时仍用7%判断
- 301588在0422涨6.29%，扫到当天满足6%门槛，但函数内部认为"不够7%不算启动日候选"
- 最终结果：函数返回score=1（融资-45.3%+无背离），正确被2分门槛过滤

**修复内容**：
- `find_true_launch_date`签名加`min_rise_pct=7.0`参数
- 4处`>= 7`替换为`>= min_rise_pct`
- `analyze_stock_v2`签名加参数透传
- 主循环调用时传入`min_rise_pct`

**验证**：
- 0422扫301588：启动日0422✓，pre15=0天✓，pre8干净✓，但融资-45.3%（爆降）→ 总分=1→排除
- 修复后-t 6行为正确

## 重要教训

### 教训1：不能凭记忆/推理回答，必须先查数据再下结论
**事件**：用户问301588为什么0422扫不到，我未查数据就说"框架只看扫到当日涨幅是否达标，0422当天301588未达6%门槛"。这是**完全错误的想当然回答**。

**正确姿势**：
- 用户质疑数据时 → 立即查Tushare验证
- 任何结论性陈述前 → 必须有数据支撑
- 不能说"应该是"/"大概"/"我记得"

### 教训2：异常排查要从数据入口trace，不能猜
**事件**：加debug后发现函数运行正常但结果为空，猜测是merge/格式/过滤条件问题，绕了一大圈。

**正确姿势**：
- 异常时先查异常处理（except Exception: continue）是否静默吞掉了
- 逐步打印关键变量值，找到第一个不符合预期的位置
- 数据类bug要在脚本里加临时print，不要在外部重新写测试脚本

### 教训3：301588在0422确实涨6.29%，但总分=1分被正确排除
**实测数据**：
- 启动日：0422 ✓
- pre15异动：0天 < 4天 ✓
- pre8干净：0414~0421，0天异动 ✓
- 融资变化：**-45.3%**（爆降！）→ 融资得分=0
- 背离：无（0415~0421方向混乱，涨跌幅都小）
- 总分：0+0+1(量比)+0(融资)=**1分 < 2分门槛**
- 象限：D（不符合A/B/Baux/C）

**结论**：301588被排除是因为基本面不合格（融资爆降），不是因为扫描器有bug。框架行为正确。

### 教训4：-t参数影响的是"扫到股票的门槛"，不是"启动日识别门槛"
**澄清**：-t 6和-t 7的区别：
- 7%门槛：扫到当天涨幅必须>=7%，find_true_launch_date内部也用7%找启动日
- 6%门槛：扫到当天涨幅>=6%即可，find_true_launch_date内部用6%找启动日
- **之前混淆**：以为-t只影响扫到门槛，不影响启动日识别，导致301588在-t 6时被我误判为"能扫到"
